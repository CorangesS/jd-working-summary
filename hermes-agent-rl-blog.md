# 从电商长程任务到 Agent RL：我如何改造 Hermes，并用 slime 构建可训练的 Agent Loop

> 这篇文章记录了我在电商采销与供应链场景中改造 Hermes Agent 的过程：为什么最初选择 Hermes、为什么单纯依靠 Prompt 和 Workflow 无法解决长程任务问题、为什么我最终基于 slime 手写 Agent Loop，并尝试通过 GRPO 优化长轨迹中的效率、稳定性和准确率。
>
> 文中涉及的时间和准确率数字应以实际实验记录为准。这里重点讨论问题结构、技术设计和方法论。

## 一、任务背景：为什么电商场景需要一个长程 Agent

跨境电商的采销、商品管理和供应链工作，并不是简单的“问答任务”。

以 Joybuy 的业务为例，一个采销人员每天需要处理：

- 竞品类目分析
- 商品机会发现
- 商品寻源
- 商品发品
- PDP 优化
- 库存诊断
- 智能补货
- 进销存计划
- PO 审核
- 文件生成和业务系统操作

这些任务往往横跨多个系统，依赖大量外部和内部数据，并且需要连续调用多个工具。

一个典型任务可能需要：

```text
理解用户目标
→ 查询内部数据
→ 抓取竞品信息
→ 清洗和归一化字段
→ 调用类目预测
→ 查询业务规则
→ 校验结果
→ 生成报告
→ 生成文件
→ 返回 AGUI 组件
→ 必要时执行系统操作
```

单个复杂任务通常需要 20～50 次工具调用，批量任务则可能进一步扩展到数百个子步骤。

这类任务的难点，不在于模型能不能回答某个问题，而在于它能否：

- 长时间保持任务目标一致
- 正确拆解任务
- 选择合适的工具
- 判断哪些步骤可以并行
- 处理工具失败
- 维护业务状态
- 避免重复调用
- 控制 Token 和延迟
- 输出符合前端协议的结果

因此，我需要的不是一个普通聊天机器人，而是一个真正具备长程执行能力的企业级 Agent。

## 二、为什么最初选择 Hermes

项目初期，我们决定使用 Hermes 作为 Agent 基座。

原因并不复杂：Hermes 已经具备较好的通用指令理解、工具调用和多轮交互能力，能够作为电商垂直 Agent 的基础模型。

相比从零搭建，使用 Hermes 可以更快进入业务改造阶段：

- 增加电商采销知识
- 接入内部和外部工具
- 增加商品、类目、供应链相关 Skill
- 接入 OpenAI 风格的 Tool Calling
- 支持 AGUI 和结构化输出
- 支持文件、报告和资源链接返回

在初期实验中，Hermes 能够较好完成单步或短链路任务。例如：

```text
输入商品链接
→ 抓取商品信息
→ 预测类目
→ 补齐属性
→ 生成发品内容
```

这让我们一度认为，继续增加 Prompt、工具和 Workflow，就可以覆盖全部业务任务。

但当任务从单步扩展到长程、从单商品扩展到批量任务后，问题开始集中暴露。

## 三、改造 Hermes 后发现的核心问题

### 3.1 单任务能完成，不代表批量任务能扩展

一个非常典型的 Case 是商品发品。

人工发品一个商品可能需要约 30 分钟，Agent 可以将单商品发品压缩到约 5 分钟。

从单商品角度看，收益已经很明显。

但是，当任务变成“批量发品 100 个商品”时，执行时间并没有接近理想的并行时间。

理想情况下，如果大部分商品可以并行处理，那么 100 个商品的总耗时应该接近：

```text
最慢单商品执行时间
+
批次调度开销
+
最终校验与提交时间
```

也就是说，理论上可能控制在约 10 分钟。

但实际运行中，100 个商品可能需要约 34 分钟，而且结果并不总是稳定。

这说明 Agent 虽然学会了“如何发一个商品”，但并没有学会：

- 如何识别批处理任务
- 如何对商品分组
- 如何构建执行 DAG
- 哪些步骤应该并行
- 哪些步骤必须串行
- 如何复用公共规则
- 如何共享中间结果
- 如何隔离失败商品
- 如何局部重试
- 如何控制并发和限流
- 如何在批量任务中保持字段不串扰

本质上，这是一个规模泛化问题。

模型在训练中见过单商品发品，却没有见过 100 商品、多阶段、多并发、部分失败的复杂发品轨迹。

### 3.2 类目规划任务：会分析，但不会高效组织分析

类目规划可能包含：

```text
获取竞品类目树
→ 获取 Joybuy 类目树
→ 完成类目映射
→ 识别未覆盖类目
→ 抓取竞品商品
→ 抽取关键属性
→ 划分价格带
→ 生成机会格子
→ 筛选机会商品
→ 推荐供应商
→ 生成分析报告
```

问题在于，Agent 经常采用局部 ReAct 模式：

```text
调用一次工具
→ 看结果
→ 再决定下一次
→ 再调用一次
```

这种方式在探索任务中很灵活，但在长程分析中会产生几个问题：

- 过多串行调用
- 重复搜索同类信息
- 对多个站点逐个处理，而不是并行处理
- 每轮重新阅读大量历史信息
- 计划随执行逐渐漂移
- 报告结论与工具证据不完全一致
- 已经获得足够信息后仍继续搜索

最终结果可能是“能做完”，但耗时、Token 和工具调用成本过高。

### 3.3 商品寻源任务：工具路径不稳定

商品寻源需要综合：

- 商品品牌
- 商品类目
- 目标国家
- Dealer 信息
- 国内供应商信息
- 品牌官网
- 第三方渠道
- 供应商资质
- 历史合作信息

Agent 可能出现：

- 一开始选择公开搜索
- 后来切换到内部数据库
- 再回到公开搜索
- 重复查询相同品牌
- 没有复用前面已获得的供应商信息
- 多个来源冲突时不会稳定裁决

这类任务的核心不是“有没有答案”，而是“是否使用了稳定、低成本、可审计的寻源路径”。

### 3.4 智能补货任务：状态和计算链容易漂移

补货任务通常依赖：

```text
历史销量
+ 当前库存
+ 在途库存
+ 销售计划
+ 安全库存
+ 补货周期
+ 季节因素
+ 异常事件
```

它并不是一次模型推理，而是一条状态敏感的计算链。

长程执行中容易出现：

- 时间窗口变化
- 商品 ID 混用
- 库存口径不一致
- 销量单位不一致
- 使用了旧的中间数据
- 一部分商品计算成功，另一部分被遗漏
- 最终解释和实际计算结果不一致

因此，补货任务不仅需要模型推理，还需要严格的业务状态管理和过程校验。

### 3.5 PO 审核任务：自由 Agent 不够可靠

PO 审核涉及：

- 字段完整性
- 供应商资质
- 数量限制
- 价格规则
- 成本规则
- 预算规则
- 风险规则
- 人工复核条件

这类任务如果完全交给自由 ReAct Agent，可能出现：

- 跳过关键审核步骤
- 工具调用顺序错误
- 在证据不足时直接通过
- 把不可自动审核的情况当成普通异常
- 最终意见与规则结果不一致

因此，PO 审核更适合：

```text
SOP 状态机
+
LLM 处理非结构化信息
+
规则引擎执行确定性校验
```

而不是纯粹依靠模型自由规划。

### 3.6 输出格式问题：业务正确不等于任务成功

Hermes 的输出并不只是自然语言。

复杂任务可能要求：

- Deep Research 风格报告
- 指定格式的 Markdown
- AGUI 组件
- 商品表格
- 图表配置
- 文件资源
- OSS URL
- `<resource>` 标签
- 可继续交互的 Action ID

例如系统要求：

```xml
<resource>https://oss.example.com/report.xlsx</resource>
```

即使模型生成的 URL 正确，只要没有被 `<resource>` 包裹，前端也可能无法识别。

因此，任务成功包含两层：

```text
业务结果正确
+
输出协议正确
```

这也是普通问答训练很难覆盖的问题。

## 四、为什么仅靠 Prompt 和 Workflow 不够

面对这些问题，第一反应通常是：

- 优化 System Prompt
- 增加 Few-shot
- 增加任务 SOP
- 增加 Workflow
- 增加工具描述
- 增加格式约束

这些方法确实有效，但只能解决一部分问题。

### Prompt 可以解决什么

Prompt 适合解决：

- 明确任务目标
- 告诉模型工具边界
- 指定输出格式
- 注入业务规则
- 约束基本行为

### Workflow 可以解决什么

Workflow 适合解决：

- 固定流程
- 强约束执行顺序
- 必须经过的审核节点
- 高风险业务操作
- 可审计流程

### Prompt 和 Workflow 解决不了什么

它们不擅长直接解决：

- 在多个可行计划中选择更优计划
- 学会何时并行
- 学会减少重复工具调用
- 在长轨迹中保持稳定
- 根据失败动态调整策略
- 控制总 Token 和总延迟
- 学会及时停止
- 在任务规模变化时调整执行方法

这些问题更像“策略优化”，而不是静态规则补充。

因此，我开始考虑通过强化学习优化 Hermes 的长程行为。

## 五、为什么使用 slime

我最终选择 slime，并不是因为它在理论上一定优于所有 RL 框架。

最直接的原因是：

> 我对 slime 非常熟悉。

在工程项目中，框架选择不应该只看论文指标，也要考虑团队的实际掌握程度。

我熟悉 slime 的：

- Rollout 数据流
- 自定义 `generate()` 接口
- Reward 接入方式
- SGLang 推理路径
- Megatron 训练路径
- GRPO 训练流程
- Token、Logprob 和 Loss Mask 组织方式

这让我能够把主要精力放在 Hermes 的 Agent 行为上，而不是重新学习一个训练框架。

这是一个合理的工程选择，但它并不意味着 slime 是唯一选择，也不意味着它天然适合所有 Agent RL 任务。

## 六、我如何基于 slime 改造 Search-R1

slime 中的 Search-R1 是一个非常清晰的多轮 RL 示例。

它的核心循环是：

```text
LLM 输出 <search>
→ 执行搜索
→ 返回 <information>
→ 再次调用 LLM
→ 最终输出 <answer>
```

它的优点是轨迹完全 Append-only：

```text
Prompt
+ Assistant Search
+ Search Observation
+ Assistant Search
+ Search Observation
+ Final Answer
```

模型生成 Token 参与训练，工具 Observation Token 不参与训练。

但 Hermes 比 Search-R1 复杂得多：

- 使用 OpenAI 风格 Tool Calling
- 一轮可能有多个 Tool Call
- Tool Result 需要通过 `tool_call_id` 关联
- 不同模型使用不同 Chat Template
- 需要保持结构化 Messages
- 需要支持 AGUI 和资源协议
- 需要在 SGLang 推理和 Megatron 训练之间保持 Token 一致

因此，我没有简单复制 Search-R1，而是计划实现一个 `search-r1-for-Hermes`。

## 七、Search-R1-for-Hermes 的核心设计

### 7.1 双轨迹设计

我保留两套轨迹。

#### 语义轨迹

```text
messages
├── system
├── user
├── assistant
├── assistant.tool_calls
└── tool results
```

用于：

- 工具执行
- Tool Call 关联
- 业务审计
- 状态分析
- Debug
- 最终输出构建

#### Token 轨迹

```text
rollout_token_ids
rollout_log_probs
rollout_loss_mask
```

用于：

- RL 训练
- Token 精确对齐
- 记录模型真实动作
- 记录真实推理前缀

Token 轨迹只追加，不回写，不用重渲染后的结果覆盖历史。

### 7.2 为什么必须保留 Messages

Hermes 一轮可能产生多个并发调用：

```text
call_1 → 抓取商品 A
call_2 → 抓取商品 B
call_3 → 抓取商品 C
```

工具返回必须关联：

```text
tool_call_id = call_1
tool_call_id = call_2
tool_call_id = call_3
```

纯字符串虽然也能表示，但最终仍然需要自己重新设计一套结构化协议。

保留 Messages 可以兼容现有 Hermes Runtime 和 OpenAI Tool Calling 生态。

### 7.3 为什么不能只保留 Messages

结构化 Tool Call 重新渲染后，字节和 Token 可能变化。

模型真实生成：

```json
{"name":"web","arguments":{"q":"abc"}}
```

重新序列化后可能变成：

```json
{"arguments":{"q":"abc"},"name":"web"}
```

语义一样，但 Token 不一样。

RL 需要训练模型真实采取的动作，因此必须保留：

- 原始生成文本
- 原始生成 Token
- 解析后的 Tool Call 对象

不能只保存解析后的结构。

### 7.4 Canonical Renderer

为了保证训练和推理一致，我需要一个唯一的 Canonical Renderer。

它负责：

- System/User 消息渲染
- Tool Schema 渲染
- Assistant 边界
- Tool Result 边界
- Generation Prompt
- 特殊 Token
- Tokenizer 配置

理想情况下：

```text
同一组 input_token_ids
├── 直接发送给 SGLang
└── 直接写入 slime rollout
```

这样可以避免：

```text
SGLang 内部渲染
vs
本地 apply_chat_template
vs
Megatron 数据渲染
```

三条路径不一致。

## 八、我希望通过 RL 优化哪些问题

### 8.1 批量任务规划

让模型学习：

- 识别批处理任务
- 按类目、品牌或站点分组
- 构建任务 DAG
- 区分批次级和商品级任务
- 复用公共规则
- 复用查询结果

### 8.2 并行与串行决策

让模型学习：

- 无依赖步骤并行
- 有依赖步骤串行
- 避免不必要等待
- 控制并发度
- 遇到限流时动态降速

### 8.3 工具选择

让模型学习：

- 优先使用内部高质量数据
- 避免重复搜索
- 避免调用无关工具
- 根据任务阶段选择最合适工具
- 在工具失败时切换备选路径

### 8.4 错误恢复

让模型学习：

```text
参数错误 → 修正参数
工具超时 → 有限重试
数据为空 → 调整 Query
部分失败 → 继续成功任务
关键数据缺失 → 转人工复核
不可恢复错误 → 明确终止
```

### 8.5 停止策略

让模型避免：

- 证据不足时过早停止
- 任务已完成后继续调用工具
- 长时间陷入重复搜索
- 在没有状态变化时循环

### 8.6 Token 和延迟优化

让模型倾向于：

- 更短的有效计划
- 更少的重复上下文
- 更少的无效工具调用
- 更高的信息复用率
- 更合理的并行规划
- 更低的每成功任务 Token 成本

### 8.7 输出协议

让模型稳定生成：

- 合法 Tool Call
- 合法 AGUI
- 正确 Resource 标签
- 指定格式报告
- 与业务结果一致的最终输出

## 九、为什么选择 GRPO

GRPO 的吸引力在于，它不依赖单独训练一个 Value Model，而是对同一个 Prompt 采样多个候选轨迹，在组内比较 Reward。

对于 Agent 任务，一个 Prompt 可以产生多种执行路径：

```text
轨迹 A：串行调用，正确但慢
轨迹 B：并行调用，正确且快
轨迹 C：少调用工具，但结果不完整
轨迹 D：格式正确，但业务结果错误
```

通过组内比较，可以鼓励模型倾向于：

- 更正确的轨迹
- 更高效的轨迹
- 更低成本的轨迹
- 更稳定的轨迹

GRPO 的工程优势包括：

- 不需要额外维护 Value Model
- 与 slime 的现有训练路径较契合
- 适合使用规则型或程序型 Reward
- 能够比较同一任务下的多种行为策略

## 十、这些任务的 Reward 对 GRPO 友好吗

答案是：部分友好，但并不天然友好。

### 10.1 对 GRPO 友好的部分

以下指标可以程序化评估：

- 最终任务是否完成
- 商品字段是否完整
- 类目是否正确
- 工具参数是否合法
- Tool Call 格式是否合法
- AGUI Schema 是否合法
- Resource 标签是否正确
- 调用工具数量
- Token 消耗
- 总延迟
- 重复调用次数
- 并行比例
- Artifact 是否存在

这些指标相对客观，适合构造 Reward。

例如：

```text
Reward =
业务成功率
+ 格式合法率
+ 参数正确率
+ 并行效率
- Tool Call 成本
- Token 成本
- 延迟成本
- 重复调用惩罚
```

### 10.2 不友好的部分

部分任务很难得到准确 Reward。

例如品类机会分析：

- “机会格子”是否真的有商业价值
- 推荐商品是否具有长期潜力
- 供应商是否值得合作
- 报告结论是否具有洞察
- 当前建议是否能真正提升销售

这些结果可能需要：

- 业务专家评分
- 后续销售结果
- 在线 A/B 实验
- 真实采销采纳率
- 长周期反馈

Reward 会非常稀疏、延迟和带噪声。

### 10.3 长程任务的 Credit Assignment

如果最终任务失败，很难判断是哪一步导致：

- 初始计划错误
- 第 5 次工具选择错误
- 第 17 次参数错误
- 中间状态覆盖
- 最终报告格式错误

只使用终局 Reward，可能导致学习效率低。

因此需要增加过程 Reward：

- Tool Call 合法性
- 参数正确性
- 状态推进
- 子任务完成
- 无重复调用
- 失败恢复
- 中间结果质量
- 最终格式合法性

## 十一、除了 GRPO，还有哪些选择

### 11.1 SFT

SFT 仍然是必须的。

适合教会模型：

- Tool Calling 格式
- 基础任务流程
- AGUI Schema
- Resource 格式
- 常见 SOP
- 标准错误恢复动作

如果模型连协议都不稳定，直接使用 RL 会非常低效。

### 11.2 DPO / IPO / ORPO 等偏好优化

适合有成对偏好数据时：

```text
更好的轨迹
vs
更差的轨迹
```

例如：

- 并行轨迹优于串行轨迹
- 低成本轨迹优于重复调用轨迹
- 正确格式优于错误格式
- 稳定恢复优于整体失败

偏好优化通常比在线 RL 简单，但它依赖高质量 Preference Pair。

### 11.3 PPO

PPO 是经典选择，能够结合 Value Model 做更细粒度优化。

优点：

- 理论和工程经验丰富
- 可做 Token-level Advantage
- 适合更细的策略更新

缺点：

- 系统复杂
- Value Model 训练成本高
- 长程 Agent Reward 方差大
- 工程调试难度更高

### 11.4 RLOO

RLOO 通过 Leave-One-Out 基线降低方差，也适合多样本 Rollout。

它与 GRPO 类似，都可以避免传统 Value Model，但在 Advantage 估计方式上不同。

### 11.5 Rejection Sampling + SFT

先采样大量轨迹，再选择优质轨迹做 SFT。

优点：

- 稳定
- 简单
- 容易控制数据质量

缺点：

- 不是真正在线策略优化
- 对边界行为探索不足
- 容易过度拟合已有优质轨迹

### 11.6 Offline RL / Trajectory Optimization

使用已有线上日志和历史轨迹训练。

适合 Hermes，因为真实业务中可能积累大量：

- 成功轨迹
- 失败轨迹
- 工具日志
- 人工修复记录
- 用户采纳记录

但需要解决日志策略偏差和 Reward 缺失问题。

## 十二、我对当前方案的批评

下面以面试官或技术评审的视角，批评这套方案。

### 12.1 把所有效率问题都归因于模型是不准确的

100 个商品发品需要 34 分钟，不一定主要是模型策略问题。

还可能是：

- 工具本身吞吐低
- 接口有并发限制
- 爬虫资源不足
- 数据库连接池不足
- OSS 上传慢
- 发布接口串行
- Runtime 调度器效率低
- 单个工具 P95 延迟过高

如果这些问题由基础设施导致，RL 不会真正解决。

#### 改进

先拆分总耗时：

```text
总耗时 =
LLM 推理
+ Tool 执行
+ 队列等待
+ 网络
+ 重试
+ 序列化
+ 文件处理
+ 最终提交
```

明确哪些是模型决策问题，哪些是系统工程问题。

### 12.2 模型不应该直接负责线程调度

模型可以决定：

- 哪些任务独立
- 哪些任务有依赖
- 哪些任务可以并行

但真正的：

- 线程池
- 协程
- 限流
- 队列
- 超时
- 重试
- Backoff
- Checkpoint

应该由 Runtime 负责。

如果让模型直接决定底层调度细节，会增加不确定性。

#### 改进

模型输出 DAG 或批处理计划，Runtime 确定性执行。

### 12.3 GRPO 可能优化出 Reward Hacking

如果 Reward 强调：

- 少调用工具
- 低 Token
- 低延迟

模型可能学会：

- 跳过必要验证
- 过早结束
- 输出看似完整但证据不足的结果
- 把复杂任务转为失败或人工复核
- 生成格式正确但业务质量低的结果

#### 改进

使用硬约束和分层 Reward：

```text
先满足业务正确性阈值
再优化效率
```

不能让效率奖励抵消业务失败。

### 12.4 组内相对奖励可能掩盖整体质量低

GRPO 比较同一组内的多个轨迹。

如果一组轨迹都很差，只是其中一个“相对没那么差”，它仍然可能得到正向 Advantage。

#### 改进

增加绝对门槛：

- 业务成功率阈值
- 格式合法阈值
- 关键步骤完成阈值
- 安全规则硬约束

低于阈值的整组轨迹不应被当作有效正样本。

### 12.5 Reward 可观测性不足

很多业务任务无法立即得到真实 Reward。

例如：

- 机会商品最终是否卖得好
- 供应商是否可靠
- 补货建议是否真正准确
- PDP 是否提高转化

训练时只能使用代理指标。

#### 改进

明确区分：

```text
训练代理指标
线上真实业务指标
```

并定期检查两者是否相关。

### 12.6 长程 Agent 的核心可能是状态管理，而不是 RL

如果 Agent 经常：

- 混淆商品 ID
- 丢失中间状态
- 使用旧数据
- 忘记已完成步骤

优先应该改 Runtime：

- Typed State
- Checkpoint
- State Store
- Variable Binding
- Tool Result Reference
- Context Compression

不能期待 RL 自动补偿缺失的状态基础设施。

### 12.7 Prompt Token 优化不应该只靠模型学习

如果每轮把全部历史、全部工具和全部业务规则塞进 Prompt，Token 一定很高。

这应通过架构解决：

- 动态工具选择
- Context Window 管理
- 状态摘要
- 结果引用
- Tool Result 压缩
- 分层 Prompt
- 缓存公共上下文

RL 可以进一步优化，但不是第一责任人。

### 12.8 缺少严格的 Baseline

如果只说：

```text
RL 后更快、更准
```

说服力不够。

需要比较：

- 原始 Hermes
- Prompt 优化后的 Hermes
- Workflow 版本
- SFT 版本
- SFT + GRPO
- 不同 Reward 版本
- 不同并发策略
- 不同任务规模

#### 改进

建立统一 Benchmark：

```text
1 / 5 / 10 / 20 / 50 / 100 商品
```

并衡量：

- 成功率
- 完整率
- P50/P95 延迟
- Tool Call 数
- Token 数
- 重试数
- 并行率
- 格式合法率
- 人工介入率

## 十三、我会如何进一步改进

### 13.1 先完成系统归因

将失败分为：

```text
模型规划错误
工具选择错误
参数错误
工具系统错误
调度错误
状态错误
格式错误
业务规则错误
```

不同错误使用不同方法解决。

### 13.2 分阶段训练

推荐：

```text
协议 SFT
→ 单任务 SFT
→ 批量轨迹 SFT
→ Preference Optimization
→ GRPO
→ 在线评估
```

不要一开始就让 RL 从随机行为中学习完整长程任务。

### 13.3 Curriculum Learning

从小规模逐步扩大：

```text
1 商品
→ 5 商品
→ 10 商品
→ 20 商品
→ 50 商品
→ 100 商品
```

同时逐步加入：

- 部分失败
- 限流
- 数据缺失
- 工具超时
- 混合类目
- 动态重规划

### 13.4 分层 Reward

```text
Level 1：协议合法
Level 2：工具调用正确
Level 3：子任务完成
Level 4：最终业务正确
Level 5：效率优化
```

先保证正确，再优化效率。

### 13.5 模型与 Runtime 分工

模型负责：

- 任务理解
- 计划
- 工具选择
- DAG 生成
- 异常决策
- 最终表达

Runtime 负责：

- 并发执行
- 状态存储
- 变量绑定
- 限流
- 重试
- 缓存
- 格式校验
- Artifact 管理

### 13.6 训练和线上一致性

必须保证：

- 同一 Tokenizer
- 同一 Chat Template
- 同一 Tool Schema
- 同一特殊 Token
- 同一 JSON 序列化约定
- 同一输出 Validator

否则训练出的策略在线上可能失效。

## 十四、面试时应该如何表达这项工作

不要把它简单描述为：

> 我用 GRPO 微调了 Hermes。

更准确的表达是：

> 我负责将 Hermes 从单步工具调用助手改造成适用于电商采销和供应链场景的长程 Agent。过程中，我发现单任务能力无法自然扩展到批量、多工具和长轨迹任务，因此基于 slime 自定义 Agent Rollout Loop，维护结构化 Messages 和 Token-exact Rollout Ledger，并设计任务完成、工具效率、格式合法性和执行成本等多维 Reward，通过 SFT 与 GRPO 优化批量规划、并行调用、错误恢复和停止策略。同时，我将并发、状态、重试和 Schema 校验下沉到 Runtime，避免把确定性系统问题错误地交给模型学习。

这个表达比“做了 Prompt 优化”或“用 RL 提升 Agent”更能体现：

- 业务理解
- Agent 架构能力
- RL 工程能力
- 系统归因能力
- 训练与推理一致性意识
- 对模型和 Runtime 边界的理解

## 十五、总结

电商长程 Agent 的难点，不是让模型学会调用某一个工具，而是让它在复杂任务中持续做出正确决策。

Hermes 初步解决了单任务能力，但在批量发品、类目规划、商品寻源、补货和 PO 审核等场景中，暴露出：

- 长程执行慢
- 并行规划不足
- 工具调用重复
- 状态容易漂移
- 失败恢复不稳定
- Token 成本高
- 输出格式不稳定
- 单任务能力难以扩展到批量任务

我选择 slime，主要是因为我对它足够熟悉，能够快速自定义 Rollout 和 Reward，而不是因为它是唯一正确的框架。

GRPO 适合用来比较同一任务下的多条执行轨迹，并优化策略效率，但它不是万能方案。

真正有效的系统需要同时具备：

```text
高质量 SFT
+
可验证 Reward
+
GRPO 或其他策略优化
+
稳定的 Agent Runtime
+
严格的状态管理
+
Token-exact 训练推理一致性
+
可靠的工具与业务评估体系
```

最终目标不是让模型“更像一个会思考的人”，而是让 Hermes 成为一个：

> 能够稳定、快速、低成本地完成真实电商业务任务，并且其行为可验证、可审计、可扩展的企业级 Agent。
